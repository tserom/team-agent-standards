# 回复末尾：修改的文件

完成**涉及改动的实现、修复、重构**后，在回复**末尾**增加独立小节 **`### 修改的文件`**（无改动则省略）。

## 格式

用 Markdown 表格，两列：

| 文件 | 说明 |
|------|------|
| `相对路径/文件名` | 一句话说明本文件改了什么（业务/行为向，少写函数名堆砌） |

- 路径用仓库内**相对路径**，行内用反引号包裹。
- 说明每条一行，聚焦「为什么 / 用户可见行为」，不必罗列每个函数。
- 若用户关心提交状态，可附一句：`共 N 个文件`；有未提交改动时可写「当前工作区有修改、尚未 commit」（仅在确实查过 `git status` 或用户问提交时说）。

## 示例

### 修改的文件

| 文件 | 说明 |
|------|------|
| `src/components/Foo/index.tsx` | 确定时回传维度类型 |
| `src/pages/Bar/services/barApi.ts` | 按类型分支调用接口 |

共 **2** 个文件。

## 何时省略

- 仅答疑、评审、读代码，**未改任何文件**。
- 用户明确只要结论、不要变更清单。

## 与其它约定

- 正文仍按用户规则写清「改了什么、为什么」；本表是**路径索引**，不替代说明。
- 需要时可在改完后执行 `git status -s` 核对列表，避免漏文件。
