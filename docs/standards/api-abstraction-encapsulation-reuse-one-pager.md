# API 分层决策 · 一页纸速查（v1.0）

> 完整版：`api-abstraction-encapsulation-reuse.md` · 适用各项目 `*Api.ts` / `services`

---

## 30 秒记住

| | 做什么 | 怎么做 |
|---|--------|--------|
| **复用** | 算数、解析 list、拼参 | **L0**，不发 HTTP |
| **封装** | 一个 URL | **L1**，你平时的习惯，**必做** |
| **封装** | 一个用户动作 | **L3**，一个 export + 步骤注释 |
| **抽象** | 多步流程 | **L2**，**少做**；副作用必须完全一致 |

**口诀**：URL 必封；动作单独函数；编排别万能。

---

## 四层（从上到下）

```
L3 场景入口    submitXxx / changeYyy     ← 产品一句话 = 一个函数
L2 场景编排    syncAandB (慎用)          ← ≥3 个入口步骤+副作用都相同才抽
L1 单 URL      postAdjustBatch           ← 每个 URL 一个（必须）
L0 工具        normalizeList / calcTotal ← 随意复用
```

## 新需求：5 问（命中即停）

| # | 问 | 是 → |
|---|-----|------|
| 1 | 只拼参/解析/计算？ | **L0** |
| 2 | 只调 **1 个** URL？ | **L1** |
| 3 | 新的用户动作？ | **新建 L3**（别塞进旧 `sync*`） |
| 4 | 已有 ≥3 个 L3，步骤和副作用 **完全相同**？ | 才考虑 **L2** |
| 5 | 会比别的场景 **多/少调** 某个 URL？ | **禁止**扩 L2 → **新建 L3** |

**编排骨架**：主表总量 **变不变**？要不要 **补偿类接口**（saveOrUpdate / rebalance）？  
→ 不变且不要补偿 → **不要** `sync` + **不要** 默认 rebalance。

---

## 必须 / 禁止

| ✅ MUST | ❌ MUST NOT |
|---------|-------------|
| 所有 URL 进 `*Api.ts` | UI/列里手写 adjust + PATCH 串联 |
| 每动作一个 L3 + 步骤注释 | 为单场景给 L2 加第 3 个 flag |
| 维护入口表（动作→函数→URL） | L0 里发 request |
| 新 L2 写清 **会调 / 不会调** 的 URL | 「凡改子表都走同一个 sync」 |

---

## 该拆了（任意一条）

- 传 `options` 要靠读长注释  
- 新场景 = 给 L2 加 `if (purpose === …)`  
- 抓包出现 PRD **没要求** 的接口  
- 文档要写「除了 A 场景外…」  
- 失败刷新策略不同却共用一个函数

|------|-----|------|
